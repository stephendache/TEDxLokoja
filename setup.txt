# Why you are unable to login after hosting on AWS

When you host a Firebase-backed application on a new domain (like an AWS Amplify URL, an S3 website URL, or a custom domain), Firebase Authentication will block login attempts from that domain by default for security reasons.

To fix this, you need to add your new AWS domain to the **Authorized Domains** list in your Firebase project.

### How to fix the login issue:

1. Go to the [Firebase Console](https://console.firebase.google.com/).
2. Select your project.
3. In the left-hand menu, click on **Authentication**.
4. Go to the **Settings** tab (or **Settings -> Authorized domains** depending on the console version).
5. Scroll down to the **Authorized domains** section.
6. Click the **Add domain** button.
7. Enter the exact domain where your app is hosted on AWS (e.g., `main.d123456789.amplifyapp.com` or `www.yourcustomdomain.com`). 
   *Note: Do not include `https://` or any trailing slashes `/`.*
8. Click **Add**.

Once you add the domain, it might take a few minutes for the changes to propagate. After that, Google Sign-In and other Firebase Authentication methods will work perfectly on your AWS-hosted site.

# Why are Firebase config keys public? Can I hide them in `.env`?

It is a very common misconception that the keys in `firebase-applet-config.json` (like `apiKey`, `projectId`, `authDomain`) need to be kept secret. 

**The short answer is: They are supposed to be public.**

### Why are they public?
These keys are simply **identifiers**, not secret passwords. They tell your frontend React code *which* Firebase project it needs to connect to. Because your React code runs in the user's browser, the browser needs these identifiers to know where to send login requests or database queries.

### Will putting them in a `.env` file hide them?
**No.** If you move them to a `.env` file (e.g., as `VITE_FIREBASE_API_KEY`), Vite will simply bundle those values directly into your compiled JavaScript files during the build process. Anyone who inspects your website's source code will still be able to see them. 

### How do I actually secure my database?
Since the config keys are public, you secure your Firebase project using **Firebase Security Rules** (the `firestore.rules` file). 

Security Rules run on Google's servers and check every single request. Even if a hacker has your `apiKey` and tries to write to your database, the Security Rules will block them unless they are properly authenticated and authorized.

### How to use `.env` for environment management (Optional)
While `.env` won't *hide* the keys from the browser, it is still useful if you want to switch between a "Development" Firebase project and a "Production" Firebase project.

If you want to do this:
1. Create a `.env` file in your project root:
   ```env
   VITE_FIREBASE_API_KEY=your_api_key
   VITE_FIREBASE_AUTH_DOMAIN=your_auth_domain
   VITE_FIREBASE_PROJECT_ID=your_project_id
   ```
2. Update your `src/firebase.ts` to use these variables instead of the JSON file:
   ```typescript
   const firebaseConfig = {
     apiKey: import.meta.env.VITE_FIREBASE_API_KEY,
     authDomain: import.meta.env.VITE_FIREBASE_AUTH_DOMAIN,
     projectId: import.meta.env.VITE_FIREBASE_PROJECT_ID,
     // ... other config
   };
   ```
*(Remember: Even after doing this, the keys will still be visible in the browser's network tab and source code, which is completely safe and expected!)*
